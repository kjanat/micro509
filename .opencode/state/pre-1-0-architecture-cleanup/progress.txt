# Progress Log

PRD: pre-1-0-architecture-cleanup
Started: 2026-03-13

## Codebase Patterns

- Keep top-level public modules easy to discover, but move complex engines behind nested internal modules when the domain seam is real.
- Parse boundaries should reject malformed input early and loudly; typed result unions are for expected operational failures, not raw byte-structure rejection.
- Cert and CSR built-in extensions should share one definition source for OID, encode, decode, and parse-assignment semantics; verifier support remains a separate concern.
- Policy and name-constraints already behave like isolated engines inside `verify.ts`; the right move is extraction, not another layer of wrapper helpers around the monolith.
- DER hardening belongs in one strict spine. Avoid ad hoc length parsing, TLV rebuilding, or child-walk helpers spread across parse domains.
- Domain-owned public types are part of the API. Shared type buckets like `validation.ts` should be temporary at most.
- Result harmonization should preserve domain meaning by moving stateful outcomes into `value`, not by flattening everything into one generic error enum.
- Shared result primitives can land incrementally: first adopt structural base error interfaces in existing domains, then flip public unions to `Result<T, E>` in later tasks without creating import cycles.
- Top-level raw-DER parsers should enter through `readSequenceChildren(..., { maxDepth: DEFAULT_MAX_DER_DEPTH })`; nested helpers can then parse exact slices without reintroducing trailing-data acceptance.
- Type re-home works best as a two-step move: make the new domain module the source of truth first, then keep the old shared file as a pure re-export shim until later cleanup removes it entirely.
- When a public file name already occupies the obvious directory path (`verify.ts`), extract internal engines as sibling `*-engine.ts` modules instead of forcing a disruptive facade rename just to get nested paths.
- For extracted validators, keep the engine failure shape structurally compatible with the facade's failure union when possible; that lets the facade delegate without inventing another translation layer.
- Registry foundations can land before the big rewire if the registry is fully typed and test-covered, while existing parser/builder entrypoints still use their old control flow.

## Task - [result-model-foundation]

- Added shared public and internal result primitives in `src/result.ts` and `src/core/result.ts`.
- Reused the new base error shapes in `src/verify.ts`, `src/identity.ts`, and `src/revocation.ts` without forcing the larger public result migration yet.
- Exported the shared result primitives from `src/index.ts` and added a dedicated `result` build entry in `tsdown.config.ts`.
- Files changed: `src/core/result.ts`, `src/result.ts`, `src/verify.ts`, `src/identity.ts`, `src/revocation.ts`, `src/index.ts`, `tsdown.config.ts`, `.opencode/state/pre-1-0-architecture-cleanup/prd.json`, `.opencode/state/pre-1-0-architecture-cleanup/progress.txt`
- **Learnings:** base error interfaces are a low-risk bridge: they make later API unification easier without forcing every caller to migrate in the same commit.

## Task - [der-spine-hardening]

- Hardened `src/der.ts` to reject unsafe integer inputs, non-minimal long-form lengths, overflowed DER lengths, trailing root data, and overly deep nesting.
- Updated top-level certificate, CSR, CRL, OCSP, PFX, and PKCS#7 parsers to enter through stricter sequence parsing with a shared max-depth guard.
- Tightened `childrenOf()` parent-boundary checks in `src/asn1.ts` so malformed child lengths cannot silently overrun a container.
- Added regression coverage for non-minimal lengths, trailing DER data, and max-depth failures in `test/internals.test.ts` and `test/malformed-der.test.ts`.
- Files changed: `src/der.ts`, `src/asn1.ts`, `src/parse.ts`, `src/crl.ts`, `src/ocsp.ts`, `src/pfx.ts`, `src/pkcs7.ts`, `test/internals.test.ts`, `test/malformed-der.test.ts`, `.opencode/state/pre-1-0-architecture-cleanup/prd.json`, `.opencode/state/pre-1-0-architecture-cleanup/progress.txt`
- **Learnings:** the safest incremental spine change is to make the root parse stricter first, then let internal exact-slice parsers stay small; this catches hostile input without forcing a full parser rewrite in one step.

## Task - [extract-policy-engine]

- Rewired `src/verify.ts` to build policy state via `createPolicyValidationState()` and delegate policy evaluation to `evaluatePolicyChain()`, then map engine failures back onto the existing verify failure shape.
- Kept current policy behavior and public outputs intact while removing the inline policy graph implementation from `src/verify.ts`.
- Files changed: `src/policy-engine.ts`, `src/verify.ts`, `.opencode/state/pre-1-0-architecture-cleanup/prd.json`, `.opencode/state/pre-1-0-architecture-cleanup/progress.txt`
- **Learnings:** extracting the engine behind one state-constructor + one evaluator keeps the orchestration seam small and preserves existing tests while the public API is still in flux.

## Task - [extract-name-constraints]

- Rewired `src/verify.ts` to build name-constraint state via `createNameConstraintValidationState()` and delegate evaluation to `evaluateNameConstraints()`.
- Kept the extracted engine failure shape compatible with existing verify failures, so `verify.ts` can return engine failures directly without behavioral churn.
- Files changed: `src/name-constraints-engine.ts`, `src/verify.ts`, `.opencode/state/pre-1-0-architecture-cleanup/prd.json`, `.opencode/state/pre-1-0-architecture-cleanup/progress.txt`
- **Learnings:** a direct engine result shape works well for refactors like this when the facade already owns the public error contract and the engine only needs a strict subset of it.
