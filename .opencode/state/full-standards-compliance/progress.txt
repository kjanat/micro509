## Codebase Patterns

- OCSP `ResponderID` parsing should preserve the CHOICE branch in parsed output; accept the normal implicit `byKeyHash` bytes, but keep parser coverage for explicit wrapper variants used in hand-built fixtures.
- Delta CRL support should validate the complete/delta pair up front (issuer, AKI, IDP scope, numbering), then overlay certificate-specific delta entries on the complete-CRL view; `removeFromCRL` only clears a prior `certificateHold`.
- Indirect CRL revocation lookup is sequence-scoped state, not entry-local state: carry `certificateIssuer` forward across revoked entries, match on serial plus effective issuer, and fail closed only after scanning all same-serial candidates.
- Structured distribution-point builder coverage should hit both optional halves of the RFC model: issuer-only certificate DPs and scope-only CRL IDPs, plus runtime rejects for empty `fullName` sequences.
- For structured distribution-point coverage, prefer builder-backed round trips that assert both certificate-side DPs and CRL-side IDP/freshest-CRL parsing, with at least one `relativeName` case and one `fullName`+reasons case.
- For RFC 9618 mappings, keep graph mutation keyed by mapped `expectedPolicySet`, then derive caller-visible policy success from the surviving root-domain policies that still reach the leaf; never let an intermediate `anyPolicy` bridge satisfy arbitrary caller policy OIDs by itself.
- For partially supported RFC surfaces, parser owns preserving/classifying unsupported wire forms; validator owns fail-closed decisions. Never silently drop unsupported critical semantics in parsing.
- ESM-only, WebCrypto-only, platform-neutral core; avoid Node builtins in core paths.
- TypeScript rules are strict: no `any`, no non-null assertions, no `as Type`, no classes.
- Tabs, single quotes, `.ts` import extensions, readonly-heavy interfaces, discriminated unions.
- Keep path validation, service identity, and revocation as separate public concepts.
- Put forward-looking cross-domain types in `src/validation.ts`, but keep verify-owned inputs narrowed to the identity forms `verify.ts` actually enforces today.
- New domain boundaries can ship as root-exported source modules first; keep broader shared unions in `src/validation.ts`, and fail closed inside the new boundary for not-yet-wired identity forms.
- Package exports are driven by `tsdown.config.ts`; do not hand-edit generated `package.json` exports.
- For stable package subpaths, prefer a named tsdown `entry` object; let `exports: true` derive package exports from entry keys.
- Adding a new public package subpath is a tsdown config change first: add the source module to the named `entry` map, rebuild, and let generated `package.json` exports update.
- In tsdown multi-entry builds, `unbundle: true` preserves shared internal modules under `dist/src/` while keeping public entry files at `dist/<entry>.js`; stable package exports can stay pointed at the root entry files.
- Root `micro509` export should stay ergonomic; granular domains should live behind stable subpath exports.
- New boundary modules should define neutral shared-source aliases locally; avoid leaking CRL/OCSP-branded source types into orchestration APIs.
- Distribution-point model change is a clean break; no temporary URI-only compat alias.
- RFC 5280 `directoryName` comparison is semantic: ordered DN, unordered RDN sets, RFC 4518-prepared AVA strings; never compare raw DER prefixes.
- RFC 9618 policy validation should use a depth-indexed DAG keyed by `(depth, validPolicy)` plus constrained policy sets; never build or expose the deprecated RFC 5280 policy tree during core validation.
- CRL applicability needs structured certificate DP + CRL IDP objects; URI-only projections are debug/display data, not enough for RFC 5280 scope decisions.
- Standards docs should keep README + `docs/PKIX-SCOPE.md` aligned around a short `implemented` / `partial` / `not yet` matrix, then narrow per-feature wording to shipped behavior.
- New verify-surface knobs should land as direct optional fields on input objects, then flow through wrapper helpers unchanged until enforcement work is ready.
- When a verify-surface task adds another RFC input family, widen every verify/profile input interface plus the shared forwarding helper in one pass so wrappers stay aligned with raw path validation.
- When RFC validation knobs are public before enforcement exists, normalize them into one internal state object and fail closed with typed "not implemented yet" results instead of silently ignoring them.
- New extension families should land in three steps: add stable OIDs + public data shapes first, then builder/parser wiring, then validation/enforcement. Do not add `CertificateExtensionsInput` or parsed-certificate fields before encode/decode exists.
- For staged RFC extensions, let builders own structural RFC rejects early (empty sequences, `anyPolicy` misuse, required critical encodings) even before parse/validation work lands.
- For CRL scope surfaces, replace URI shortcut fields in one clean break with structured distribution-point objects, then keep encode/parse helpers aligned on `fullName` vs `relativeName` and reason-mask semantics.
- When moving a concern behind a new boundary module, keep verify/profile APIs stable by adapting boundary-specific failures at the composition seam instead of duplicating domain logic.
- For profile helpers that combine purpose and service identity, run chain/purpose validation first, then call the identity boundary so EKU/profile failures do not get masked by hostname mismatches.
- When parser coverage needs parsed-only unsupported RFC forms, keep builder-facing extension types generic with a supported default and widen only parse-owned outputs; that preserves encode type safety while letting validators fail closed.
- `directoryName` general names are IMPLICIT constructed values: builders must encode only the inner SEQUENCE content, and validators must compare parsed RDN structure plus prepared strings instead of raw DER.
- Treat caller-supplied initial name constraints as the first accumulated constraint level, then layer certificate `nameConstraints` on top; that preserves RFC intersection semantics and existing self-issued exemptions.
- For staged extension families, parser helpers should mirror builder guardrails on empty/repeated/forbidden RFC structures, and preserve custom `ANY` payloads as raw DER so round-trip-enough typed data lands before enforcement logic.
- For staged standards work, land internal state machines before public enforcement, but keep public policy/revocation knobs fail-closed until the matching enforcement/output task exists.
- For staged policy enforcement, enable only the RFC knobs the current task actually enforces, keep the rest fail-closed, and derive pass/fail from the final constrained policy set before exposing richer public outcomes.
- Expose policy outcomes as two views: `authorityConstrainedPolicies` reports the final surviving leaf-domain policies, while `userConstrainedPolicies` reports caller-visible policy results after mapping and `anyPolicy` handling.
- CRL handling should split signer/freshness validation from certificate applicability: first validate issuer, AKI, signature, time, and `cRLSign`; then return explicit `non_applicable` reasons for scope mismatches instead of treating a fresh wrong-scope CRL as usable evidence.
- Revocation orchestration should normalize each evidence source into `good` / `revoked` / indeterminate results, short-circuit on any revocation, and preserve per-source indeterminate details so mixed CRL/OCSP evidence never fails open.
