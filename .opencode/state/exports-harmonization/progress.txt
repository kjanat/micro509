# Progress Log

PRD: exports-harmonization
Started: 2026-03-15

## Codebase Patterns

- Root-barrel cleanup can de-headline advanced tuning/plumbing first while leaving heavily-used workflow exports in place until the broader docs and export-map cutover.
- Scaffold future public domains as pure re-export barrels first; that creates stable landing zones before risky file moves.
- Public entrypoints should follow product domains, not source filenames.
- `micro509` is the canonical documented import; domain entrypoints are the exhaustive advanced homes.
- Each public symbol has one owner domain; root re-exports are convenience only.
- Land API ownership inventories as spec artifacts before file moves; they make later export and import edits mechanical instead of guesswork.
- `src/internal/**` is the only implementation-only tree; helpers and engines should move there aggressively.
- Public leaf modules may import internals, but must not import public barrels.
- Tests that need internals should use an explicit `#micro509/internal/*` path, not incidental public seams.
- `tsdown.config.ts` should be the source of truth for public entrypoints; JSR exports should be generated from the same entry map.
- `micro509/pem` stays public as a narrow PKI boundary module.
- PKCS#12 MAC support stays under `pkcs`, but should be documented as advanced plumbing, not a headline surface.
- Source files should reach `src/internal/**` through relative imports so build-time `.ts` rewriting still works; tests can keep `#micro509/internal/*` aliases.
- `@internal` jsdoc can still affect emitted declarations for exported helpers, so new canonical domain barrels should not promote symbols that declaration emit already treats as internal.
- When a utility domain becomes canonical, point the build entry at its domain barrel and move shared cross-domain primitives under `src/internal/shared/**` instead of leaking them through a neighboring public module.
- Canonical domain barrels can tighten value exports, but should preserve legacy type re-exports during the migration so the advanced import home grows without breaking compile-time callers.
- Multi-file workflow domains can become canonical by routing root exports and build entries through one curated barrel while legacy leaf subpaths stay as compatibility seams for a later cutover.
- Keep the canonical public entry map in `tsdown.config.ts`; let tsdown own npm exports and update only `jsr.json`'s `exports` field from a post-build hook.
- When `tsdown.config.ts` imports manifest JSON directly, `tsconfig.other.json` needs `resolveJsonModule` plus a JSON include, and the post-build hook should mutate only the JSR fields it owns (`exports`, version).
- Once root and domain barrels cover the public workflows, shrink the generated export map to owner domains only and scrub docs/examples/tests of file-shaped import seams at the same time.
