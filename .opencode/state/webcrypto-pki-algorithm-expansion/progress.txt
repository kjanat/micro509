## Codebase Patterns

- Keep signature-algorithm work split cleanly: `src/parse.ts` preserves wire data, `src/signing.ts` owns emission, `src/sig-verify.ts` owns acceptance, and `src/keys.ts` owns WebCrypto key import/generation mapping.
- For partially supported RFC surfaces, parser preserves unsupported-but-well-formed forms; validator and builder decide the shipped subset and fail closed outside it.
- New algorithm support should stay WebCrypto-first and platform-neutral: no Node-only crypto branches in core paths.
- Backward compatibility matters at API boundaries: prefer optional fields like RSA `scheme` and explicit builder `signature` inputs over breaking existing call patterns.
- When widening algorithm surface, update all four touchpoints together: `src/oids.ts` for identifiers, `src/keys.ts` for public WebCrypto-facing inputs, `src/signing.ts` for builder-side mapping, and `src/sig-verify.ts` for verifier-side mapping.
